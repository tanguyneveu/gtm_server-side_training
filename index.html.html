Le DataLayer

L'objet JavaScript qui structure et fiabilise toutes vos données de tracking

DataLayer

Le DataLayer : La vérité terrain

Définition technique

Un objet JavaScript (dataLayer = []) invisible qui stocke les données structurées de votre site de manière standardisée et fiable.

Pourquoi est-il vital ?

✓
Complètement indépendant du design HTML/CSS
✓
Langage commun entre développeurs et marketing
✓
Assure la fiabilité des données e-commerce
✓
Pérennité face aux évolutions du site
DataLayer

Le DataLayer en pratique

Pourquoi utiliser le DataLayer ?

1

Fiabilité

Évite de "scraper" le contenu visuel du site qui peut changer et casser le tracking.

2

Richesse des données

Remonte des infos invisibles à l'écran : ID utilisateur, marge produit, type de client.

3

Standardisation

Langage commun lisible par tous vos outils marketing sans recoder les intégrations.

DataLayer

Structure et Push DataLayer

Règle d'or : placé avant le code GTM pour les données statiques
dataLayer = [{
  'pageCategory': 'Blog',
  'visitorType': 'Client'
}];

Le DataLayer fonctionne par paires Clé / Valeur.

Variables de couche de données

Dans GTM, on configure des variables pour "lire" ces valeurs. Ex: ecommerce.value ou ecommerce.transaction_id

DataLayer

Structure et Push DataLayer

Règle d'or : Ne jamais écraser — toujours utiliser .push()
window.dataLayer.push({
  'event': 'purchase',
  'ecommerce': {
    'transaction_id': 'T12345',
    'value': 120.00,
    'currency': 'EUR',
    'items': [{
      'item_name': 'Produit Premium',
      'price': 120.00
    }]
  }
});

Variables à lire

Dans GTM, créez des variables de couche de données pour lire :

→
ecommerce.value
→
ecommerce.transaction_id
→
ecommerce.items.0.item_name
DataLayer

Du DataLayer vers GTM

1

Variable de couche de données

Dans GTM, on crée une variable qui pointe vers une "clé" du DataLayer (ex: price). GTM lit automatiquement la valeur associée (ex: 50.00).

2

Déclencheur d'événement personnalisé

GTM s'active dès qu'il détecte un mot-clé via dataLayer.push.

dataLayer.push({
  'event': 'purchase',
  'transactionId': '12345',
  'transactionTotal': 120.00
});
DataLayer

Intégration

1

Avec les plugins de CMS

Les CMS proposent des plugins de DataLayer e-commerce standardisés (ex: Addingwell pour Shopify, WooCommerce, PrestaShop...).

Solution rapide et no-code recommandée pour les plateformes e-commerce.
2

Manuellement

Le DataLayer peut être intégré manuellement dans le code du site en JavaScript, en coordination avec les développeurs.

Nécessite une spec de DataLayer rédigée par le tracking analyst et remise aux développeurs.

Le Tracking
Server-Side

Nouveau paradigme, architecture et mise en place technique

Contexte

Pourquoi le tracking traditionnel s'essouffle ?

Performance dégradée

Scripts JS lourds exécutés dans le navigateur, ralentissant l'expérience globale.

AdBlockers omniprésents

~1/3 des Français utilisent un adblocker. Certains bloquent même le chargement de GTM.

ITP & restrictions cookies

Safari/iOS : cookies limités à 24h (gclid/fbclid) ou 7 jours max sur tous appareils Apple.

Vulnérabilités sécurité

Clés API et données sensibles exposées dans le code source du navigateur.

Résultat : fin programmée des cookies tiers, perte massive de signal publicitaire et baisse significative du ROAS.
La solution

Le Tracking Server-Side : La solution

Le nouveau paradigme

Principe fondamental : Ajouter un serveur intermédiaire (proxy) que vous contrôlez totalement entre le navigateur et les plateformes tierces.

Navigateur — Utilisateur
↓
Votre Serveur — tracking.monsite.com
↓
Plateformes — Google, Facebook, etc.

Les bénéfices immédiats

✓
Cookies first-party longue durée (13 mois)
✓
Contournement des AdBlockers
✓
Contrôle total des données envoyées
✓
Sécurité renforcée (clés API côté serveur)
✓
Amélioration des performances Web (Core Vitals)
Architecture

Architecture GTM Server-Side

Nouveaux concepts essentiels

1

Le Client

Écoute les requêtes HTTP entrantes et les traduit en Event Data exploitable. La "porte d'entrée" du serveur.

2

Event Data

Objet de données standardisé interne au serveur. Similaire au DataLayer mais côté serveur.

3

Tags Server-Side

Transmettent les données depuis le serveur vers les API officielles des plateformes marketing.

Analogie : Le Client = le réceptionniste. L'Event Data = le formulaire rempli. Les Tags = le courrier envoyé aux destinataires.

Meta CAPI

Meta Conversions API (CAPI)

Tracking Facebook/Instagram côté serveur

Qu'est-ce que la Meta CAPI ?

API officielle de Meta pour envoyer des événements de conversion directement depuis votre serveur — sans dépendre du Pixel navigateur.

Le Pixel Meta est bloqué par ~30% des utilisateurs. iOS 14.5+ a drastiquement réduit le signal. La CAPI complète ou remplace ce signal.

Configuration dans GTM Server

1
Installer le tag "Facebook Conversions API"
2
Renseigner votre Pixel ID et Access Token
3
Mapper : event_name, value, currency, email hashé...
4
Configurer la déduplication via event_id
Déduplication

Éviter les doublons : Pixel + CAPI

Le risque : Pixel ET CAPI envoient purchase → Meta comptabilise 2 conversions au lieu d'1.

La solution — l'event_id

1
Générer un ID unique par événement : purchase_1234567
2

ITP

Cookies First-Party : Le problème ITP

Qu'est-ce qu'ITP ?

Intelligent Tracking Prevention — technologie Apple (Safari) qui limite automatiquement la durée de vie des cookies JavaScript.

Impact : un utilisateur Safari qui achète 8 jours après son premier clic → conversion non attribuée.

Durées de vie des cookies

TypeNavigateurDurée max
Cookie JS avec gclid/fbclidSafari/iOS24 heures
Cookie JS sans paramètresSafari/iOS7 jours
Tout cookie JSFirefox/Brave7 jours
Cookie HTTP first-partyTous13 mois
Solution

Cookies First-Party : La solution server-side

Comment ça fonctionne

Votre serveur dépose le cookie via la réponse HTTP (en-tête Set-Cookie). Ce cookie est "first-party" car il vient de ss.monsite.com.

Navigateur → ss.monsite.com
↓
Serveur : Set-Cookie: _ga=...; max-age=34128000
↓
Cookie _ga valide 13 mois même sur Safari ✓

Résultat

Cookie JS (client-side)24h–7j (ITP)
Cookie HTTP first-party13 mois
Le cookie _ga persiste 13 mois sur Safari → attribution et segmentation correctes sur tout le funnel.

✓
